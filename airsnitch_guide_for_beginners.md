# AirSnitch 초보자 메뉴얼 (네트워크 클라이언트 격리 취약점 점검을 위한 안내서)

본 매뉴얼은 Wi-Fi 보안 기술과 클라이언트 격리(AP 격리) 취약점에 대한 전문적인 배경 지식이 부족한 일반 관리자 및 시스템 운영자분들도 AirSnitch 도구를 활용하여 사내망 또는 가정용 망의 안전성을 직접 테스트할 수 있도록 작성되었습니다.

## 1. 개요 및 목적
AirSnitch는 무선 공유기(AP)에 연결된 각 사용자(기기) 간의 통신을 물리적, 논리적으로 차단하는 '클라이언트 격리' 기능이 실제 제대로 작동하는지 검사하는 점검 도구입니다. 메인 네트워크와 게스트 네트워크가 완벽히 분리되어 동작하는지 판별하는 데 사용하십시오. 이 도구는 실제 비밀번호를 해킹하는 것이 아닌, 공유기가 패킷을 전달하는 방식상의 허점을 찾는 데 사용됩니다.

## 2. 사용 시 주의 사항 (필독)
이 도구를 실행하기 전에 다음 사항들을 반드시 숙지하십시오.

1. 타인의 네트워크 금지: 오직 자신이 합법적인 소유 권한을 가진 네트워크와 장비만을 대상으로 점검을 수행하십시오. 권한 없이 타인 또는 외부망을 점검하는 것은 불법이며 보안 침해에 해당할 수 있습니다.
2. 실 서비스망 영향도 우려: 점검 중 공유기와 네트워크에 순간적인 트래픽 오류나 접속 끊김(간섭) 현상이 발생할 수 있습니다. 사내 중요한 통신망은 일과 중에 피하시고, 독립적인 테스트 환경에서 먼저 시연하십시오.
3. 운영체제 버전 제한: 이 도구는 반드시 'Ubuntu 22.04' 버전의 리눅스 환경과 USB 무선 랜카드(Wi-Fi 동글) 두 개 이상이 필요합니다. Windows 운영체제에서는 정상적으로 동작하지 않습니다.

## 3. 사전 준비 (설치 방법)
이 과정은 한 번만 수행하면 됩니다.

1. 인터넷 터미널(명령 프롬프트)을 열고, 다음 명령어를 순서대로 복사하여 붙여넣기 한 뒤 Enter 키를 눌러 필수 프로그램을 설치합니다.
   sudo apt update
   sudo apt install libnl-3-dev libnl-genl-3-dev libnl-route-3-dev libssl-dev libdbus-1-dev pkg-config build-essential net-tools python3-venv aircrack-ng rfkill git dnsmasq tcpreplay macchanger

2. 설치 파일이 있는 디렉토리로 이동한 후, 다음 명령어들을 한 줄씩 실행하여 내부 프로그램을 셋업합니다.
   ./setup.sh
   cd airsnitch/research
   ./build.sh
   ./pysetup.sh

## 4. 점검 전 필수 설정 (매번 실행)
프로그램을 시작할 때 항상 아래 환경을 먼저 준비해야 합니다.

1. 리눅스 시스템의 일반 Wi-Fi 접속 관리자(Network Manager)를 비활성화하거나 꺼두십시오. 이 기능이 켜져있으면 충돌하여 테스트가 작동하지 않습니다.
2. 루트 디렉토리에서 아래 명령어로 관리자 권한을 획득하고 파이썬 가상 공간으로 진입합니다.
   cd airsnitch/research
   sudo su
   source venv/bin/activate
3. 테스트 대상을 지정하기 위해 `research` 폴더 안에 있는 `client.conf` 파일을 메모장(또는 텍스트 편집기)으로 엽니다. 다음과 같이 피해자와 공격자의 Wi-Fi 정보를 각각의 실제 환경에 맞게 키보드로 수정하여 저장하십시오.

   network={
       id_str="victim"
       ssid="테스트할 메인 네트워크의 Wi-Fi 이름"
       key_mgmt=WPA-PSK
       psk="메인 네트워크의 Wi-Fi 비밀번호"
   }

   network={
       id_str="attacker"
       ssid="테스트할 게스트 네트워크의 Wi-Fi 이름"
       key_mgmt=WPA-PSK
       psk="게스트 네트워크의 Wi-Fi 비밀번호"
   }

## 5. 본격적인 취약점 점검 실행하기 (사용법)
준비가 끝났다면 다음 3가지 핵심 테스트를 차례대로 명령어를 입력하여 점검합니다. 테스트 명령어 내의 `wlan2` 및 `wlan3`는 USB로 장착하신 랜카드 장치명에 따라 `wlan0`, `wlan1` 등으로 다르게 기입하셔야 할 수도 있습니다. 양쪽 네트워크의 이름이 다르면 반드시 `--no-ssid-check` 를 붙여야 합니다.

(1) 암호키 강제 유출-공유 테스트 (GTK 그룹 키 악용)
게스트와 일반 사용자가 동일한 암호키를 나눠가지며 격리가 깨져 있는지 봅니다.
명령어:
   ./airsnitch.py wlan2 --check-gtk-shared wlan3 --no-ssid-check --same-bss

결과 판단:
화면에 'The victim's GTK is' 와 'The attacker's GTK is' 뒤에 표시되는 문자열이 똑같다면 취약한 상태입니다.

(2) 공유기 반송 스푸핑 테스트 (Gateway Bouncing)
공유기가 중간에서 통신을 거부하지 않고 L2 계층 주소를 변조한 공격자 패킷을 그대로 타겟에게 전달해 버리는지 봅니다.
명령어:
   ./airsnitch.py wlan2 --c2c-ip wlan3 --no-ssid-check --same-bss

결과 판단:
'Client to client traffic at IP layer is allowed' 라는 빨간색 문구가 표시된다면 취약한 상태입니다. 분리가 정상적이라면 진행이 일정 시간 멎거나 에러가 뜹니다.

(3) 내부 포트 스틸링 테스트 (통신 가로채기)
외부 서버 응답을 공격자 자리로 라우팅시켜 빼돌릴 수 있는지 조사합니다. 외부 인터넷과 통신이 잘 되는지 확인하기 위한 핑 통신 서버가 필요합니다.
명령어:
   ./airsnitch.py wlan2 --c2c-port-steal wlan3 --no-ssid-check --other-bss --server 8.8.8.8

결과 판단:
'Downlink port stealing is successful.' 이라는 빨간 문구가 뜬다면 포트 스틸링 가로채기에 무방비하게 뚫린 상태입니다.

## 6. 결론 및 사후 대책
위의 점검 명령어 중 하나라도 '취약' 상태를 가정한 경고가 나온다면 귀하의 네트워크 구성품(라우터/공유기)은 완벽한 클라이언트 격리를 제공하지 못하고 있다는 뜻입니다. 

문제를 자체적으로 해결하기 위해서 다음 방법을 고려해 보십시오.
- 제조사 펌웨어 최신 업데이트: 제조사 측에 해당 취약점이 패치된 새 최신 펌웨어가 있는지 확인 후 장비를 업데이트 하십시오.
- 올바른 VLAN 단절: 방화벽 옵션에서 물리적 MAC 분리 뿐만 아니라 IP 계층까지 통신 차단벽 룰을 올바르게 체크하여 구성했는지 다시 매뉴얼을 살피십시오.
- 클라이언트 격리 비활성화 안내 제공: 공용 개방형 Wi-Fi 서비스라면 본 옵션에 한계가 있음을 이용자들에게 경고하고 인터넷에 민감한 작업을 조심하라 공지해야 합니다.
  


## 7. 부록: 취약점이 발생하는 원리 (비전문가용 해설)

도구가 점검하는 3가지 공격 기법이 도대체 어떤 허점을 찌르는 것인지, 복잡한 통신 용어를 배제하고 알기 쉽게 설명해 드립니다.

### (1) GTK 그룹 키 악용: "전체 방송용 봉투에 개인 편지를 몰래 넣기"
* **정상 상황**: 공유기는 공지사항이나 전체 방송(브로드캐스트)을 보낼 때, 접속한 모든 사람이 열어볼 수 있는 공용 암호키(GTK)를 사용합니다.
* **취약점 작동 방식**: 공격자는 이 '공용 암호키'로 암호화된 전체 방송 패킷 안에, 특정 한 사람(피해자)만 읽을 수 있는 '개인 편지(유니캐스트 IP 패킷)'를 몰래 숨겨서 공유기에 던집니다. 공유기는 전체 방송인 줄 알고 모두에게 전달해 버리며, 피해자의 기기는 그 안에 든 편지가 자신을 향한 것이란 걸 알고 열어보게 됩니다. 결국 차단벽을 무시하고 직접 통신이 이루어진 셈입니다.

### (2) 게이트웨이 바운싱: "우체국장(라우터)을 속여서 수신자 조작하기"
* **정상 상황**: 기기 간 격리가 켜져 있으면, 사용자 A와 사용자 B는 공유기(스위치)단에서 서로 연락을 주고받지 못합니다.
* **취약점 작동 방식**: 네트워킹에는 눈에 보이는 겉봉투(MAC 주소)와 진짜 목적지(IP 주소)가 있습니다. 공격자는 겉봉투의 도착지를 '공유기(우체국장)'로 적어 보냅니다. 그러면 공유기는 "아 나한테 온 거구나!" 하고 통과시켜 줍니다. 그런데 공유기가 봉투를 열어 속내용(IP 주소)을 보니 진짜 목적지는 '피해자 B'로 적혀 있습니다. 스위치의 단순 차단 기능만 있는 공유기(라우터)는 친절하게도 이 편지를 B에게 다시 반송(바운싱)해서 전달해 버립니다.

### (3) 포트 스틸링: "거짓 이사 신고로 우편물 가로채기"
* **정상 상황**: 공유기나 내부 연결 스위치는 각 사용자가 통신망의 어느 번호표(포트)에 있는지 장부(MAC 테이블)에 기록해 둡니다.
* **취약점 작동 방식**: 공격자가 갑자기 관리자(게이트웨이)나 피해자의 번호표(MAC 주소)를 위조하여 네트워크에 접속합니다. 그러면 내부 스위치 장부는 "어? 이 사람이 저쪽 자리로 이사 갔네?" 하고 잘못된 기록을 덮어쓰게 됩니다. 그 결과 피해자에게 가야 할 우편물(다운링크)이나 피해자가 밖으로 보내는 우편물(업링크)이 엉뚱하게 위조된 공격자 자리로 계속 배달되어 통신 내용을 훔쳐볼 수 있게 됩니다.
